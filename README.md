# Guardian of Truth — старт решения хакатон-задачи

## 1. Что это за задача и что я уже сделал

Это задача бинарной классификации: по паре `(prompt, model_answer)` нужно оценить вероятность фактологической галлюцинации.

Я изучил:
- постановку в `sber.md`;
- структуру публичного набора `knowledge_bench_public.csv`;
- baseline-ноутбук `baseline.ipynb` (схема: признаки из hidden states + легковесный классификатор).

### Ключевые ограничения из задания
- Главная метрика: **PR-AUC**.
- Требование к скорости: порядка **500 ms на один ответ** (с учетом правил из условия).
- Нельзя обучаться на публичном preview как на train; он используется как public test/dev для отладки.
- Разрешен (и отдельно оговорен) 1 forward pass `ai-sage/GigaChat3-10B-A1.8B-bf16` для снятия внутренних сигналов.

По `knowledge_bench_public.csv`:
- Всего примеров: **1044**.
- Баланс классов:
  - `True` (галлюцинация): **566**
  - `False` (без галлюцинации): **478**
- Колонки:
  - `prompt`
  - `model_answer`
  - `is_hallucination`
  - `correct_answer`
  - `comment`

Оценка длины текстов (в символах):
- Prompt: mean ~102.7, median 98, p90 153.
- Answer: mean ~192.2, median 124, p90 339.

Вывод: класс-перекос умеренный, поэтому PR-AUC корректная целевая метрика; ответы иногда длинные, значит у модели должны быть устойчивые фичи для длинных хвостов.

## 3. Архитектура моего решения (production-oriented)

Я предлагаю **двухконтурную систему** с приоритетом скорости:

## Контур A (основной, быстрый): Internal + Uncertainty

Используем один forward pass базовой LLM и извлекаем признаки:

1. **Token-level uncertainty фичи**
   - средний/минимальный logprob по токенам ответа;
   - перцентили logprob (p10, p25, p50);
   - доля «низкоуверенных» токенов (logprob ниже порога);
   - mean entropy (если доступно распределение).

2. **Layer-wise internal фичи**
   - hidden state статистики на нескольких слоях (например `[0, 5, 10, 15, 20, 25]`, как в baseline-подходе);
   - L2-нормы, дисперсии, cos-сходство между соседними слоями;
   - «дрейф» репрезентации по мере углубления слоя.

3. **Consistency-фичи без внешнего API**
   - простая внутритекстовая проверка чисел/дат/единиц (regex + правила);
   - бинарные фичи: есть ли конфликт чисел между вопросом и ответом;
   - доля named entities в ответе, не встречающихся в prompt (эвристика риска).

4. **Легковесный мета-классификатор**
   - CatBoost / LightGBM / LogisticRegression (на практике начну с логистической регрессии для интерпретируемости и скорости);
   - выход: `hallucination_score` в [0,1].

Почему этот контур: максимально быстрый, детерминированный, без внешних API, легко уложить в latency budget.

## Контур B (опциональный, «точность по требованию»)

Для ответов из «серой зоны» (например, score в диапазоне 0.45–0.60):
- запускаем более дорогую проверку на уровне claim-ов (локальная NLI/верификация, если укладывается по времени и ресурсам);
- итоговый скор объединяем с контуром A (stacking / calibrated blending).

Важно: Contour B можно отключить для жесткого SLA, а архитектуру оставить совместимой.

## 4. Что я считаю «моим решением» на этом этапе

На старте я фиксирую **реалистичную стратегию итераций**:

### Этап 1 (MVP, сейчас)
- Реализовать end-to-end pipeline:
  - загрузка данных;
  - extraction фичей из 1 forward pass;
  - обучение легкого классификатора;
  - оценка PR-AUC + latency per sample.
- Добавить кросс-валидацию и сохранение артефактов (`features.parquet`, `model.pkl`, `metrics.json`).

### Этап 2 (усиление качества)
- Feature engineering:
  - дополнительные uncertainty фичи;
  - контраст prompt↔answer (семантическая дистанция, overlap фичи).
- Калибровка вероятности (Platt / isotonic) для стабильного порога в проде.

### Этап 3 (оптимизация времени)
- Пакетный инференс и профилирование узких мест.
- Кэширование токенизации/служебных вычислений.
- Абляции: удаление дорогих фич с малым вкладом в PR-AUC.

### Этап 4 (hardening)
- Тесты на краевые случаи:
  - даты, числа, единицы измерения;
  - короткие/длинные ответы;
  - очень уверенная ложь vs неуверенная правда.
- Готовность к закрытому финальному бенчмарку.

---

## 5. Как будем измерять качество и скорость

1. **Качество**
   - основная: PR-AUC;
   - дополнительные: ROC-AUC, precision@fixed_recall, recall@fixed_precision.

2. **Скорость**
   - среднее/медиана/p95 latency на 1 пример;
   - отдельно замеряем:
     - время фиче-экстракции;
     - время классификатора;
     - total pipeline time;
   - в отчете явно фиксируем, что входит/не входит в замер согласно правилам задачи.

3. **Репродьюс**
   - фиксируем seed, версии библиотек, конфиги.

## 7. Что делать дальше прямо сейчас

1. Зафиксировать окружение (Python + зависимости).
2. Вынести baseline из ноутбука в `src/`-скрипты.
3. Реализовать extraction + train + eval CLI-команды.
   Пример запуска обучения:

   ```bash
   python -m src.train --data-path knowledge_bench_public.csv
   ```

   По умолчанию `--data-path` также равен `knowledge_bench_public.csv`.
4. Получить первую baseline-метрику PR-AUC и latency.
5. Запустить абляции фичей и улучшать trade-off «качество/скорость».

Если нужно, в следующем шаге я сделаю уже **кодовую реализацию MVP-пайплайна** (не только документацию): скрипты, конфиги, артефакты и шаблон отчета с метриками.
